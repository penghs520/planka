# planka 系统架构

## 项目概述

planka 是一个企业级敏捷研发管理平台，采用零代码架构设计，支持：

- **灵活的数据建模**：自定义__PLANKA_EINST__、属性定义、类型继承
- **可视化关联管理**：__PLANKA_EINST__间的关联关系定义与 ER 图展示
- **多维度视图**：列表视图、看板视图、自定义筛选条件
- **价值流管理**：支持价值流状态定义与流转
- **业务规则引擎**：支持定时触发、卡片变更触发的自动化规则

## 服务说明

### schema-service（Schema 定义服务）

元数据管理服务，负责所有配置定义的 CRUD 和生命周期管理：

- **属性定义**：文本、数字、日期、枚举、架构、附件等多种类型
- **__PLANKA_EINST__**：特征类型、__PLANKA_EINST__、继承关系
- **关联类型**：__PLANKA_EINST__间的关联关系定义
- **视图配置**：列表视图、看板视图的定义
- **菜单配置**：导航菜单结构
- **详情模板**：卡片详情页布局配置
- **创建页模板**：卡片创建页布局配置
- **价值流定义**：状态流转配置
- **业务规则**：自动化规则配置（触发器、条件、动作）
- **卡片动作**：自定义按钮动作配置
- **权限配置**：卡片权限配置
- **公式定义**：时间计算、数值计算、卡片聚合等公式

### card-service（卡片数据服务）

卡片数据管理服务，通过 zgraph-driver 与图数据库交互：

- **卡片 CRUD**：创建、查询、打开详情页、更新（卡片和关联）、移动卡片状态、丢弃、归档、还原以及对应的批量操作等
- **条件查询**：支持复杂条件、分页、排序
- **权限控制**：创建、更新与查询等操作的灵活权限控制

### view-service（视图数据服务）

视图数据聚合服务，负责视图数据的查询和转换：

- **视图执行**：根据视图定义执行数据查询
- **条件合并**：合并视图条件与用户筛选条件
- **数据转换**：将查询结果转换为视图所需格式

### user-service（用户认证服务）

用户和组织管理服务：

- **用户管理**：注册、激活、个人信息
- **认证服务**：登录、JWT 令牌、刷新令牌
- **组织管理**：组织创建、设置
- **成员管理**：成员添加、角色变更、移除

### gateway-service（API 网关）

统一 API 入口：

- **路由转发**：将请求路由到对应微服务
- **JWT 认证**：验证请求令牌
- **CORS 配置**：跨域请求处理

### extension-service（扩展服务）

扩展功能模块，包含操作历史记录等功能：

- **操作历史**：记录卡片的所有变更操作，支持多维度查询

### oss-service（对象存储服务）

文件存储服务，支持插件化存储后端：

- **预签名上传**：安全的文件上传机制
- **文件管理**：文件元数据管理
- **插件体系**：本地存储、MinIO 存储等

## 核心设计原则

### 1. 简单三层架构

每个服务内部采用三层架构：Controller → Service → Repository → Mapper

- 业务逻辑集中在 Service 层
- 实体模型为贫血模型（仅包含数据，无业务行为）
- Repository 作为防腐层隔离数据访问细节

### 2. Schema 缓存策略

除了 `schema-service` 服务之外的模块，要求使用缓存进行查询 Schema，避免使用 FeignClient 进行远程调用：

- `SchemaCacheService`：二级缓存实现（本地 Caffeine + 分布式 Redis）
- 具体查询类：
  - `CardTypeCacheQuery`：__PLANKA_EINST__缓存查询
  - `ViewCacheQuery`：视图定义缓存查询
  - `CardFaceCacheQuery`：卡片面版模板缓存查询
  - `BizRuleCacheQuery`：业务规则缓存查询
  - `LinkTypeCacheQuery`：关联类型缓存查询
  - `ValueStreamCacheQuery`：价值流缓存查询
  - `CardActionCacheQuery`：卡片动作缓存查询
  - `CardPermissionCacheQuery`：卡片权限缓存查询
  - `FieldConfigCacheQuery`：字段配置缓存查询
  - `StructureDefinitionCacheQuery`：结构定义缓存查询
  - `MenuGroupCacheQuery`：菜单组缓存查询
  - `CardDetailTemplateCacheQuery`：详情模板缓存查询
  - `CardCreatePageTemplateCacheQuery`：创建页模板缓存查询

### 3. 卡片缓存

- `CardCacheService`：卡片基础信息二级缓存
- `CardBasicInfo`：轻量级卡片信息（ID、标题、状态等）

### 4. 统一 ID 生成

- `SnowflakeIdGenerator`：雪花算法分布式 ID 生成器
- `SchemaId` 子类：各类 Schema 定义的 ID 类型

### 5. 类型安全的属性值

- 使用 Jackson 多态序列化机制处理不同类型的字段值
- 避免使用 Object 类型，使用强类型 `FieldValue` 子类

## 图数据库（zgraph）

### 技术栈

- **Rust** 实现
- **openraft**：分布式 Raft 共识
- **RocksDB**：持久化存储引擎
- **Tokio**：异步运行时
- **gRPC**：服务间通信

### 核心模块

```
zgraph/
├── src/
│   ├── database/        # 数据库核心
│   │   ├── rdb/         # RocksDB 存储层
│   │   └── model/       # 数据模型（顶点、边、查询）
│   ├── server/          # gRPC 服务
│   ├── raft/            # 分布式共识
│   └── proto/           # Protobuf 定义
```

### Java 驱动（zgraph-driver）

- 基于 Netty 的 gRPC 客户端
- 支持顶点（卡片）和边（关联）的 CRUD 操作
- 支持复杂查询和批量操作
